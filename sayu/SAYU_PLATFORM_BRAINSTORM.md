# 사유 플랫폼 브레인스토밍

## 0. 관련 문서
- 인프라/비용 브레인스토밍: [SAYU_INFRA_COST_BRAINSTORM.md](SAYU_INFRA_COST_BRAINSTORM.md)

## 1. 이름과 방향
`사유`는 "글을 사유하는 공간"이라는 뜻으로 읽힌다. 단순한 게시판이 아니라, 각 사용자가 자기만의 생각을 쌓아두고, 공개하거나 비공개로 남기고, 서로의 글을 읽고 저장하고 구독하는 플랫폼을 지향한다.

핵심 인상은 세 가지다.

- 글의 장르가 아니라 글의 태도를 중심에 둔다.
- 보여주기보다 머무르기 좋은 공간을 만든다.
- 공개와 비공개, 공유와 소장의 경계를 사용자가 직접 조절한다.

## 2. 한 줄 정의
시, 일기, 소설, 산문, 메모를 모두 담는 개인 사유 공간형 글 플랫폼.

## 2.1 플랫폼 범위
`사유`는 웹과 앱(iOS/Android)에서 모두 동작하는 것을 기본 방향으로 한다.

- 웹: 탐색, 아카이브, 작성/편집, 운영 관리의 중심 채널
- 앱: 읽기 몰입, 빠른 작성, 알림/구독 소비 중심 채널
- 원칙: 계정, 콘텐츠, 결제, 반응 데이터는 웹/앱 간 완전 동기화

## 3. 이 플랫폼이 특별해질 수 있는 지점
일반적인 블로그나 SNS와 달리 `사유`는 "팔로워를 늘리는 곳"보다 "생각을 축적하는 곳"에 가깝다.

- 사용자는 글을 쓰는 순간부터 자신의 사유 공간을 가진다.
- 글 하나하나가 독립적인 작품이자 기록이 된다.
- 공개 범위를 세밀하게 선택할 수 있다.
- 다른 사람의 글을 저장하면서 나만의 사유 목록을 만들 수 있다.
- 구독은 사람 자체를 따라가는 기능이 아니라, 그 사람의 사유의 흐름을 따라가는 기능이 된다.

## 4. 핵심 사용자 경험
### 4.1 글쓰기
- 글의 장르를 먼저 정하지 않아도 된다.
- 제목, 본문, 태그, 공개 범위만으로 빠르게 발행 가능해야 한다.
- 글쓰기 화면은 방해 요소가 적고 몰입감이 있어야 한다.

### 4.2 사유 공간
- 각 사용자는 자기만의 공간을 가진다.
- 이 공간은 방, 서재, 기록장, 작은 도서관 같은 감각으로 설계할 수 있다.
- 글 목록은 단순 피드보다 아카이브 감성을 갖는 편이 어울린다.

### 4.3 공유와 비공개
- 전체 공개: 누구나 열람 가능
- 구독자 공개: 구독한 사람만 열람 가능
- 비공개: 작성자만 열람 가능
- 링크 공유: 특정 글만 외부에 열람 가능한 방식도 고려 가능

### 4.4 저장과 구독
- 저장은 "좋아요"보다 조금 더 진지한 개념으로 설계한다.
- 구독은 사용자 전체를 따라가되, 개별 글 단위 알림도 선택 가능하면 좋다.

## 5. 최소 핵심 기능 MVP
플랫폼화를 하려면 처음부터 너무 넓게 가기보다, 다음 기능부터 시작하는 편이 좋다.

1. 회원가입/로그인
2. 개인 사유 공간 생성
3. 글 작성/수정/삭제
4. 공개 범위 설정
5. 타인 글 열람
6. 저장하기
7. 구독하기
8. 검색과 태그 탐색

MVP에서 꼭 필요한 건 "다 쓸 수 있음"과 "내가 원하면 숨길 수 있음"이다.

## 6. 다음 단계에서 붙이면 좋은 기능
### 6.1 감성 기능
- 글 분위기 태그: 고요, 단정, 불안, 회상, 질문 등
- 오늘의 사유 문장
- 작품 카드형 미리보기
- 글 내부 줄바꿈/여백 보존 강조

### 6.2 관계 기능
- 저장한 글 모음집
- 구독자의 새 글 피드
- 작가별 사유 공간 팔로우
- 글에 대한 짧은 응답이나 메모

### 6.3 아카이브 기능
- 태그 필터
- 날짜별 탐색
- 장르별 탐색
- 글 길이별 탐색

### 6.4 확장 기능
- 비공개 노트북
- 공동 편집/합작 글
- 출판용 내보내기(PDF/EPUB)
- 작성 습관 통계

## 7. 플랫폼 톤앤매너
`사유`는 화려한 SNS보다 차분하고 정돈된 인상을 주는 편이 어울린다.

- 여백이 충분한 인터페이스
- 과한 알림 최소화
- 반응형이지만 너무 앱 같지 않은 느낌
- 글 자체가 가장 먼저 보이는 구조
- 사용자 프로필도 과시보다 정체성 중심

## 8. 수익화 아이디어
플랫폼화 이후를 생각하면 다음 정도가 현실적이다.

- 무료 + 유료 구독 모델
- 프리미엄 사유 공간 테마
- 저장/아카이브 확장 기능 유료화
- 작가 페이지 커스텀 옵션
- 출판/내보내기 기능 유료화

다만 초반에는 수익화보다 "사람이 계속 글을 쓰게 만드는 경험"이 우선이다.

## 9. 플랫폼화가 너무 큰 목표인지
결론부터 말하면, **너무 큰 목표는 아니지만 한 번에 만들 목표는 아니다.**

비개발자인 입장에서 어려운 부분은 기능 자체보다도 다음 세 가지다.

- 회원 시스템과 권한 구조
- 공개/비공개/구독/저장 같은 관계 데이터 설계
- 운영 정책과 확장 방향 정리

즉, 한 번에 완성형 플랫폼을 만들려고 하면 크다. 하지만 아래처럼 나누면 충분히 현실적이다.

1. 현재 웹사이트를 `개인 사유 아카이브`로 안정화
2. 다중 사용자 계정과 프로필 추가
3. 공개 범위와 저장/구독 기능 추가
4. 검색, 추천, 알림을 단계적으로 확장

이렇게 가면 시작은 작게, 목표는 크게 갈 수 있다.

## 10. 현실적인 권장 전략
### 전략 A. 가장 안전한 길
- 현재 사이트를 기반으로 개념 검증
- 개인 글쓰기 경험을 먼저 다듬음
- 그다음 다중 사용자 구조로 확장

### 전략 B. 조금 더 야심 있는 길
- 처음부터 플랫폼 형태의 사용자 구조를 설계
- 단, 실제 개발은 핵심 기능부터 순차 구현

개인적으로는 전략 A가 맞다. 지금 사이트는 이미 감성적 정체성이 있으므로, 이를 "사유의 첫 번째 방"으로 보고 확장하는 편이 좋다.

## 11. 다음에 정하면 좋은 것
- 사유의 핵심 한 문장 소개문
- 사용자의 공간을 무엇으로 부를지: 방, 서재, 정원, 아카이브 등
- 공개 범위의 정확한 규칙
- 저장과 구독의 차이
- 첫 화면에서 무엇을 가장 먼저 보여줄지

## 12. 메모
- `사유`는 글을 "올리는" 플랫폼보다 글을 "쌓는" 플랫폼에 가깝다.
- 블로그보다 조용하고, SNS보다 느리며, 아카이브보다 개인적이어야 한다.
- 시작은 작아도 좋다. 다만 정체성은 처음부터 선명해야 한다.

## 13. 수익모델 브레인스토밍 확장
질문한 방향처럼, 중간 수수료 모델과 유료 구독자 전용 콘텐츠 모델은 충분히 가능하다.

### 13.1 모델 A: 작가 구독형
- 독자가 특정 작가를 월 구독한다.
- 구독자 전용 글, 선공개 글, 외전, 연재 프리뷰를 제공한다.
- 플랫폼은 결제 금액에서 일정 비율을 수수료로 가져간다.

장점:
- 예측 가능한 반복 매출
- 작가의 지속 창작 동기 강화

주의점:
- 구독 해지율 관리가 매우 중요
- 작가별 가치 차이가 커지면 양극화 가능

### 13.2 모델 B: 회차/작품 단건 결제형
- 웹소설처럼 특정 연재 회차 또는 작품 단위로 결제한다.
- 플랫폼은 판매 건당 수수료를 가져간다.

장점:
- 인기 작품이 빠르게 매출을 만든다.
- 구독보다 결제 진입 장벽이 낮을 수 있다.

주의점:
- 불법 공유 대응과 DRM 정책 필요
- 환불/분쟁 정책이 필수

### 13.3 모델 C: 혼합형
- 기본은 무료 공개 + 작가 구독 + 단건 결제를 함께 운영
- 작가는 자기 전략에 맞춰 수익화 방식을 선택

장점:
- 다양한 작가군을 흡수하기 좋다.
- 독자 결제 패턴에 맞춰 유연하게 대응 가능

주의점:
- 초기에 정책이 복잡해질 수 있음
- UI가 복잡해지면 전환율 하락

## 14. 사유에 맞는 권장 순서
플랫폼 정체성을 지키려면, 바로 결제부터 붙이기보다 다음 순서가 좋다.

1. 무료 기반 커뮤니티와 아카이브 경험 완성
2. 작가 구독형 유료 전용 글 도입
3. 단건 결제는 인기 작가군이 생긴 뒤 도입
4. 마지막에 추천/랭킹/프로모션 고도화

핵심은 결제가 먼저가 아니라, 결제할 이유를 먼저 만드는 것이다.

## 15. 수수료 구조 초안
초기 가설로 아래처럼 테스트 가능하다.

- 결제 수단 수수료: 외부 PG 비용
- 플랫폼 수수료: 10%~20% 구간에서 A/B 테스트
- 정산 주기: 월 1회 또는 월 2회

작가 입장에서 가장 민감한 것은 수수료율보다 정산 투명성이다.
따라서 정산 내역 화면은 초기부터 설계해두는 편이 좋다.

## 16. 리스크와 체크포인트
- 성인물/저작권/명예훼손 등 콘텐츠 정책
- 결제 취소/환불/분쟁 처리 정책
- 유료 콘텐츠 유출 대응
- 창작자 유치 전략과 독자 유입 전략의 균형

특히 수익모델을 붙이면 운영이 곧 제품이 된다.
그래서 기능 기획과 함께 정책 문서를 같이 키워야 한다.

## 17. 반응 시스템 브레인스토밍 (미확정)
이 섹션은 아직 확정 전 아이디어 모음이다.

### 17.1 후보 모델
- 모델 A: 단순 좋아요 1개
- 모델 B: 이모티콘 반응 다중 선택(중복 선택 허용)
- 모델 C: 하이브리드(좋아요 + 감정 반응)

### 17.2 사유 톤에 맞는 가설
- 반응은 "평가"보다 "감정 전달"에 가깝게 설계한다.
- 과열을 막기 위해 반응 종류는 소수(예: 4~6개)로 제한한다.
- 반응 수 경쟁을 유도하는 랭킹 노출은 초기에 최소화한다.

### 17.3 예시 반응 세트
- 공감
- 여운
- 고요
- 응원
- 놀람

### 17.4 운영 규칙 초안
- 한 유저가 한 글에 남길 수 있는 반응 개수 제한(예: 최대 2~3개)
- 반응 취소/변경 가능
- 글 화면에는 반응 합계만 노출하고, 개인 반응 상세는 비공개

### 17.5 MVP 적용안(가설)
1. 1단계: 좋아요 + 감정 반응 4개, 유저당 최대 2개
2. 2단계: 반응 취소/변경 UX 정교화
3. 3단계: 작가용 반응 인사이트(감정 분포) 제공

## 18. 미확정 아이디어 관리 규칙
- 이 문서의 `미확정` 섹션은 대화 중 자유롭게 추가한다.
- 실제 개발 착수 전에는 `확정` 여부를 따로 표시한다.
- 확정된 항목만 별도 실행 문서(요구사항/로드맵)로 이동한다.
