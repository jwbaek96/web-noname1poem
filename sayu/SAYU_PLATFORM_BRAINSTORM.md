# 사유 플랫폼 브레인스토밍

## 1. 이름과 방향
`사유`는 "글을 사유하는 공간"이라는 뜻으로 읽힌다. 단순한 게시판이 아니라, 각 사용자가 자기만의 생각을 쌓아두고, 공개하거나 비공개로 남기고, 서로의 글을 읽고 저장하고 구독하는 플랫폼을 지향한다.

핵심 인상은 세 가지다.

- 글의 장르가 아니라 글의 태도를 중심에 둔다.
- 보여주기보다 머무르기 좋은 공간을 만든다.
- 공개와 비공개, 공유와 소장의 경계를 사용자가 직접 조절한다.

## 2. 한 줄 정의
시, 일기, 소설, 산문, 메모를 모두 담는 개인 사유 공간형 글 플랫폼.

## 3. 이 플랫폼이 특별해질 수 있는 지점
일반적인 블로그나 SNS와 달리 `사유`는 "팔로워를 늘리는 곳"보다 "생각을 축적하는 곳"에 가깝다.

- 사용자는 글을 쓰는 순간부터 자신의 사유 공간을 가진다.
- 글 하나하나가 독립적인 작품이자 기록이 된다.
- 공개 범위를 세밀하게 선택할 수 있다.
- 다른 사람의 글을 저장하면서 나만의 사유 목록을 만들 수 있다.
- 구독은 사람 자체를 따라가는 기능이 아니라, 그 사람의 사유의 흐름을 따라가는 기능이 된다.

## 4. 핵심 사용자 경험
### 4.1 글쓰기
- 글의 장르를 먼저 정하지 않아도 된다.
- 제목, 본문, 태그, 공개 범위만으로 빠르게 발행 가능해야 한다.
- 글쓰기 화면은 방해 요소가 적고 몰입감이 있어야 한다.

### 4.2 사유 공간
- 각 사용자는 자기만의 공간을 가진다.
- 이 공간은 방, 서재, 기록장, 작은 도서관 같은 감각으로 설계할 수 있다.
- 글 목록은 단순 피드보다 아카이브 감성을 갖는 편이 어울린다.

### 4.3 공유와 비공개
- 전체 공개: 누구나 열람 가능
- 구독자 공개: 구독한 사람만 열람 가능
- 비공개: 작성자만 열람 가능
- 링크 공유: 특정 글만 외부에 열람 가능한 방식도 고려 가능

### 4.4 저장과 구독
- 저장은 "좋아요"보다 조금 더 진지한 개념으로 설계한다.
- 구독은 사용자 전체를 따라가되, 개별 글 단위 알림도 선택 가능하면 좋다.

## 5. 최소 핵심 기능 MVP
플랫폼화를 하려면 처음부터 너무 넓게 가기보다, 다음 기능부터 시작하는 편이 좋다.

1. 회원가입/로그인
2. 개인 사유 공간 생성
3. 글 작성/수정/삭제
4. 공개 범위 설정
5. 타인 글 열람
6. 저장하기
7. 구독하기
8. 검색과 태그 탐색

MVP에서 꼭 필요한 건 "다 쓸 수 있음"과 "내가 원하면 숨길 수 있음"이다.

## 6. 다음 단계에서 붙이면 좋은 기능
### 6.1 감성 기능
- 글 분위기 태그: 고요, 단정, 불안, 회상, 질문 등
- 오늘의 사유 문장
- 작품 카드형 미리보기
- 글 내부 줄바꿈/여백 보존 강조

### 6.2 관계 기능
- 저장한 글 모음집
- 구독자의 새 글 피드
- 작가별 사유 공간 팔로우
- 글에 대한 짧은 응답이나 메모

### 6.3 아카이브 기능
- 태그 필터
- 날짜별 탐색
- 장르별 탐색
- 글 길이별 탐색

### 6.4 확장 기능
- 비공개 노트북
- 공동 편집/합작 글
- 출판용 내보내기(PDF/EPUB)
- 작성 습관 통계

## 7. 플랫폼 톤앤매너
`사유`는 화려한 SNS보다 차분하고 정돈된 인상을 주는 편이 어울린다.

- 여백이 충분한 인터페이스
- 과한 알림 최소화
- 반응형이지만 너무 앱 같지 않은 느낌
- 글 자체가 가장 먼저 보이는 구조
- 사용자 프로필도 과시보다 정체성 중심

## 8. 수익화 아이디어
플랫폼화 이후를 생각하면 다음 정도가 현실적이다.

- 무료 + 유료 구독 모델
- 프리미엄 사유 공간 테마
- 저장/아카이브 확장 기능 유료화
- 작가 페이지 커스텀 옵션
- 출판/내보내기 기능 유료화

다만 초반에는 수익화보다 "사람이 계속 글을 쓰게 만드는 경험"이 우선이다.

## 9. 플랫폼화가 너무 큰 목표인지
결론부터 말하면, **너무 큰 목표는 아니지만 한 번에 만들 목표는 아니다.**

비개발자인 입장에서 어려운 부분은 기능 자체보다도 다음 세 가지다.

- 회원 시스템과 권한 구조
- 공개/비공개/구독/저장 같은 관계 데이터 설계
- 운영 정책과 확장 방향 정리

즉, 한 번에 완성형 플랫폼을 만들려고 하면 크다. 하지만 아래처럼 나누면 충분히 현실적이다.

1. 현재 웹사이트를 `개인 사유 아카이브`로 안정화
2. 다중 사용자 계정과 프로필 추가
3. 공개 범위와 저장/구독 기능 추가
4. 검색, 추천, 알림을 단계적으로 확장

이렇게 가면 시작은 작게, 목표는 크게 갈 수 있다.

## 10. 현실적인 권장 전략
### 전략 A. 가장 안전한 길
- 현재 사이트를 기반으로 개념 검증
- 개인 글쓰기 경험을 먼저 다듬음
- 그다음 다중 사용자 구조로 확장

### 전략 B. 조금 더 야심 있는 길
- 처음부터 플랫폼 형태의 사용자 구조를 설계
- 단, 실제 개발은 핵심 기능부터 순차 구현

개인적으로는 전략 A가 맞다. 지금 사이트는 이미 감성적 정체성이 있으므로, 이를 "사유의 첫 번째 방"으로 보고 확장하는 편이 좋다.

## 11. 다음에 정하면 좋은 것
- 사유의 핵심 한 문장 소개문
- 사용자의 공간을 무엇으로 부를지: 방, 서재, 정원, 아카이브 등
- 공개 범위의 정확한 규칙
- 저장과 구독의 차이
- 첫 화면에서 무엇을 가장 먼저 보여줄지

## 12. 메모
- `사유`는 글을 "올리는" 플랫폼보다 글을 "쌓는" 플랫폼에 가깝다.
- 블로그보다 조용하고, SNS보다 느리며, 아카이브보다 개인적이어야 한다.
- 시작은 작아도 좋다. 다만 정체성은 처음부터 선명해야 한다.
